# 设计决策与变更规则

本文档记录容易被无意改回的系统决策。除非新的需求明确推翻某项决策，否则后续修改必须保持这些约束。

## ADR-001：Agent 使用直连 HTTP 通信

**决定**：Agent 间消息由 `DirectBus` 直接 POST 目标容器 `/message`。

**原因**：通信路径与真实网络抓包一致，减少中心代理对时延、流量和故障的干扰。

**禁止回退**：

- 不把 `services/message_bus.py` 恢复为默认 relay；
- 不让 MCP `send_message` 调用中心 `/relay`；
- 不通过名称模糊匹配目标。

## ADR-002：拓扑同时定义通信权限和网络 profile

**决定**：同一拓扑边生成对称通信矩阵，并为两端生成出站 `tc` profile。

**影响**：

- 无边即不可通信；
- 有边但网络参数全零时，只建立通信权限；
- 变更拓扑字段必须同步修改严格校验和场景文档。

## ADR-003：网络事实只来自真实 PCAP

**决定**：Agent 容器内 `tcpdump` 是网络证据来源，`network.jsonl` 是真实 packet 的结构化表示。

**禁止回退**：

- 不从 HTTP middleware、Tool 事件或 Agent 消息构造 network record；
- 不把应用层 latency/status 伪装成 TCP/IP 字段；
- 不在没有 PCAP 证据时声称存在数据包。

## ADR-004：日志分为 application、network、system

**决定**：

- `application.jsonl` = Agent 行为 + 传统应用层日志；
- `network.jsonl` = 原始网络层 + 传输层日志；
- `system.jsonl` = 平台调试日志。

**变更规则**：schema 变更采用新版本并一次性迁移 producer、consumer、测试和文档，不保留无需求的旧字段兼容。

## ADR-005：Skill 源文件与 Tool 执行权限分离

**决定**：`skill_refs` 控制源文件读取；`allowed_tools` 控制原子 Tool 执行。

**禁止回退**：

- `srv` 不读 Skill 正文；
- Adapter 不注入完整 `skill_context`；
- Skill 名不注册成 Tool；
- Skill 提及某 Tool 不自动授权。

## ADR-006：逻辑 Agent 与容器槽位分离

**决定**：容器可复用，逻辑身份由每次 `/run` 上下文决定。

**影响**：

- 日志顶层 `agent_id` 必须使用逻辑身份；
- OpenCLAW session 由逻辑 Agent + trace 隔离；
- 分配前必须 reset；
- 容器环境中的静态 ID 仅用于槽位自描述。

## ADR-007：真实后端严格失败，不静默 fallback

**决定**：配置为 OpenCLAW 或 Claude Code 时，SDK/Gateway 缺失应失败。`MOCK_LLM=1` 和显式 `direct_llm` 是独立模式。

**禁止回退**：不得把 direct LLM 当作后端缺失时的透明替代。

## ADR-008：抓包完整性属于实验成功条件

**决定**：抓包启动失败、中途退出、无有效 PCAP 或 hash/manifest 不完整都会影响实验质量。

**原因**：项目目标包括可验证的 Agent 网络流量，不能把抓包当作可选装饰。

## ADR-009：跨层关联是推断，不是因果证明

**决定**：应用事件与 packet 通过 trace、Agent 和时间窗口相关联，输出必须声明其推断性质。

## ADR-010：场景编写文档独立维护

**决定**：`scenes/` 下文档用于指导场景编写；系统架构调整默认只修改 `docs/`。

**例外**：只有当场景输入合同本身发生变化，且任务明确包含场景规范更新时，才同步修改 `scenes/` 文档。

## ADR-011：运行中的剧本资源禁止物理删除

**决定**：在当前运行时仍会按需读取剧本 Skill、场景 Tool 或其他剧本资源的架构下，删除剧本前必须查询运行占用状态。仍被运行中的仿真依赖的剧本不得物理删除。

**原因**：剧本基本结构加载到内存，不代表全部运行依赖已经脱离持久化剧本资源。删除正在使用的剧本可能导致后续 Skill 读取或 Tool 加载失败。

**禁止回退**：

- 不得仅以“剧本结构已加载到内存”为由移除占用检查；
- 不得承诺删除正在运行的原始剧本后仿真必然继续正常运行；
- 不得在没有完整运行快照的情况下允许物理删除被占用剧本。

**可替代设计**：未来若仿真启动时创建包含 Agent 定义、Skill、Tool 和其他运行资源的完整快照，并保证运行过程只访问该快照，可以通过新的设计决策替代本条约束。

## ADR-012：仿真并发上限可扩展

**决定**：当前实现的并发仿真上限为 `1`，但“单仿真”属于当前运行策略，不是永久领域约束。外部接口和目标数据模型必须保留通过 `simulation_id` 和运行引用集合扩展到多个并行仿真的能力。

**当前实现**：

- 控制面使用单一 `simulation_active`、`current_scene_name` 和当前轮次状态；
- 启动新仿真前要求当前没有运行中的仿真；
- 剧本占用结果当前最多包含一个运行引用。

**扩展要求**：

- 后续提高并发上限时，将全局状态迁移为按 `simulation_id` 索引的运行注册表；
- Agent 注册表、容器分配、通信矩阵、终止配置、日志、PCAP、manifest 和网络 profile 必须按仿真实例隔离；
- 状态查询、停止、日志与结果接口必须能够指定 `simulation_id`；
- 剧本占用查询继续返回运行引用集合，不把单一当前剧本字段作为长期接口；
- 并发上限应由配置或容量策略控制，默认值可以保持为 `1`。

**禁止固化**：

- 不得把当前单例全局状态直接定义为永久外部接口；
- 不得设计只能返回一个占用仿真的剧本删除保护合同；
- 不得在扩展多仿真时共享可变 Agent 状态、容器槽位或实验产物目录。

## ADR-013：SceneDefinition 分别保存 Skill 与 Tool 定义

**决定**：目标 `SceneDefinition` 必须分别保存 `SkillDefinition[]` 与 `ToolDefinition[]`。Agent 仅通过 `skill_refs` 和 `allowed_tools` 引用对应定义，不使用统一 `CapabilityReference` 混合表示两者。

**原因**：

- Skill 是供后端按需读取的源文件或包；
- Tool 是通过运行时注册并受执行授权控制的原子操作；
- 两者的生命周期、校验规则、权限语义和运行路径不同。

**模型要求**：

- `SceneDefinition.skills` 保存剧本可用 Skill 定义；
- `SceneDefinition.tools` 保存剧本可用 Tool 定义；
- `AgentDefinition.skill_refs` 必须引用 `skills` 中存在的 Skill；
- `AgentDefinition.allowed_tools` 必须引用 `tools` 中存在的 Tool；
- 预览结果分别返回 Skill 列表与 Tool 列表；
- 查询摘要分别统计 `skill_count` 与 `tool_count`。

**禁止回退**：

- 不得将 Skill 和 Tool 合并成统一能力定义集合；
- 不得使用同一个枚举字段区分 Skill 与 Tool 后再共享相同结构；
- 不得因为 Skill 文档提及 Tool 就自动创建 Tool 授权；
- 不得只校验 Agent 引用字符串而不校验对应定义集合是否存在。

**当前实现差距**：当前代码只在 `AgentDef` 中保存 Skill 引用和 Tool 授权，尚未在 `SceneDefinition` 中构造两类定义集合。实现该目标设计时应同步修改解析器、校验器、预览模型和测试。

## ADR-014：仿真运行配置归属于仿真实例

**决定**：`SimulationRuntimeConfig` 归属于 `SimulationRun`，不属于 `SceneDefinition`、`SceneResource` 或 `ScenePreview`。

**运行配置包括**：

- `max_rounds`；
- `stalemate_rounds`；
- `network_mode`；
- `seed`；
- 后续新增的其他仿真调度、终止或运行参数。

**模型要求**：

- `SceneDefinition` 只保存静态剧本内容；
- 创建或启动仿真时显式构造 `SimulationRuntimeConfig`；
- 运行配置按 `simulation_id` 保存；
- 仿真调度和终止逻辑只读取当前仿真实例的运行配置；
- 剧本查询与预览不得返回或修改仿真运行配置；
- 读取剧本不得设置当前剧本、当前轮次或全局终止配置。

**禁止回退**：

- 不得将 `max_rounds`、`stalemate_rounds`、`network_mode` 或 `seed` 加回 `SceneDefinition`；
- 不得在 `ScenePreview` 中增加 `runtime_config`；
- 不得把剧本元数据中的运行参数作为仿真实例的隐式权威配置；
- 不得在剧本解析函数中修改全局仿真状态；
- 不得因为当前只允许一个仿真运行，就把运行配置设计为无 `simulation_id` 所有权的全局合同。

**当前实现差距**：当前剧本加载流程仍会读取 `scenario_metadata.max_rounds` 和 `scenario_metadata.stalemate_rounds`，并写入全局终止配置，同时修改当前剧本状态。后续实现应将这些行为迁移到仿真创建和启动流程，并删除从剧本隐式覆盖运行配置的兼容逻辑。

## 通用变更规则

1. 每次新工作先阅读 `docs/README.md`、相关设计文档和代码。
2. 代码与设计文档必须在同一 PR 中保持一致。
3. 当前处于初期开发阶段：优先直接迁移，不为旧实现增加兼容分支。
4. 删除架构后，删除所有文档、compose、测试和 Dashboard 中的旧引用。
5. 新增字段、接口或后台任务时，写明所有权、生命周期、失败语义和观测方式。
6. 完成后检查 diff，确保没有修改任务范围外的 `scenes/` 文档。
7. 无异常且检查通过时，直接合并。
