# AgentNetwork 开发文档

## 1. 本地启动

```bash
docker compose config
docker compose up -d --build
```

默认服务：

- `srv`：控制面与 Web，宿主机端口 8000；
- `ag-o1`：OpenCLAW 运行槽位；
- `ag-c1`：Claude Code 运行槽位。

当前 compose 不启动 `bus`。需要零 token 演示时显式设置 `MOCK_LLM=1`。

## 2. 代码模块

```text
services/
  server.py                 FastAPI 控制面和 Dashboard
  agent_server.py           Agent 容器 HTTP 运行时
  message_bus.py            已废弃的 410 兼容桩

agent_network/
  api/
    simulations.py          scene setup/launch/stop
    agents.py               Agent 查询接口
    logs.py                 日志查询、写入、文件管理
    packets.py              PCAP 查询、分析、审计、打包
    system.py               系统信息与配置
  agent_management.py       AgentRegistry、ContainerRuntime、轮次执行
  scene_def.py              场景与 Agent 数据结构
  adapters/
    base.py                 AgentContext / AgentRunResult
    claude_code.py          Claude Agent SDK Adapter
    openclaw.py             OpenCLAW SDK Adapter
    direct_llm.py           显式直接模型能力
  comm.py                   DirectBus
  mcp_server.py             系统与场景 Tool MCP
  skill_mcp_server.py       Claude Skill 源文件 MCP
  skill_source.py           Skill 路径解析和安全校验
  network_emulation.py      Linux tc 配置
  full_packet_capture.py    容器内 tcpdump 生命周期
  real_packet_store.py      PCAP 解析、查询和分析
  log_manager.py            严格日志 schema、索引、持久化
  experiment_manifest.py    实验来源、质量审计和 bundle
```

## 3. 主要 API

### 仿真

- `POST /api/simulations/setup`
- `POST /api/simulations/launch`
- `POST /api/simulations/stop`
- `GET /api/scenes`
- `GET /api/scenes/state`
- `GET /api/scenes/{scene_name}`

### Agent 容器内部接口

- `POST /run`
- `POST /message`
- `GET /status`
- `GET /inbox`
- `POST /reset`
- `POST /capture/start`
- `POST /capture/stop`
- `GET /capture/status`
- `POST /network/configure`
- `POST /network/clear`

### 日志与数据包

- `/api/logs/*`：查询、统计、导出、下载、隐藏、删除和 ingest；
- `/api/packets/*`：查询、统计、分析、PCAP 下载、实验 manifest、质量审计、bundle 和跨层关联；
- `/ws`：Dashboard 实时状态、日志和 packet 更新。

## 4. 开发流程

1. 先阅读 `docs/README.md`、相关设计文档和实际代码。
2. 确认变更属于控制面、Agent runtime、通信、Skill/Tool、日志或网络证据中的哪一层。
3. 修改代码时同步更新生产者、消费者、测试和设计文档。
4. 早期阶段不添加无需求兼容代码；旧接口应删除或明确返回已废弃。
5. 不修改 `scenes/` 下指导场景编写的文档，除非任务明确要求。
6. 对日志、通信和网络采集的变更，优先检查是否破坏证据真实性。

## 5. 最低验证集

文档或代码变更完成后至少执行：

```bash
python -m pytest
docker compose config
```

涉及后端镜像时再执行对应构建与验收：

```bash
docker compose build ag-c1
docker compose build ag-o1
./scripts/check-openclaw-runtime.sh ag-o1
```

涉及网络仿真或抓包时验证：

- 容器有 `NET_ADMIN`、`NET_RAW`；
- `tc` 和 `tcpdump` 可用；
- launch 后每个 Agent capture 状态为 running；
- stop 后 PCAP 有效且 hash 可验证；
- `/api/packets/quality?session_id=...` 通过。

## 6. 日志变更检查

修改日志时必须同时检查：

- `log_manager.py` schema 与版本；
- 所有 emitter；
- API 查询和 ingest；
- Dashboard 字段读取；
- token usage 聚合；
- experiment audit；
- tests；
- `docs/日志与实验数据设计.md`。

禁止仅在 reader 中增加兼容分支来掩盖 producer 未迁移。

## 7. 通信与网络变更检查

修改消息或网络路径时必须同时检查：

- `DirectBus` 的 exact ID 与 allowlist；
- `agent_directory` 和 `comm_matrix` 下发；
- MCP `send_message` / `broadcast`；
- `/message` inbox 语义；
- 拓扑到 `tc` profile 的转换；
- 控制面抓包过滤；
- PCAP/manifest/quality audit；
- `docs/通信与网络仿真设计.md`。

## 8. 文档验收

- `docs/README.md` 中链接均存在；
- 设计文档不引用已经删除的 `bus` 数据面、旧 logger、旧 Skill 注入或不存在的文件；
- 新设计决策已写入 `设计决策与变更规则.md`；
- diff 中不包含未明确要求的 `scenes/` 文档修改。
