# 剧本校验待完成项

> 状态：**待完成**。本文记录当前剧本管理模块尚未实现的完整语义校验范围。完成对应代码、测试和接口状态迁移前，不得将这些能力标记为已实现。

## 1. 当前已实现的校验

当前 `SceneStorage.build_definition()` 和上传流程已经具备核心结构校验：

- 上传资源必须是 ZIP；
- 归档必须包含 `meta_and_roles.json`、`instances_and_skills.json`、`network_topology.json`；
- 三个核心文件必须能够读取为 JSON；
- `skill_refs` 必须由非空字符串组成；
- Agent 后端必须属于当前支持集合；
- `network_topology.json` 必须包含根级 `topology` 数组；
- 拓扑项必须包含 `endpoint_a`、`endpoint_b` 和 `channel_id`；
- 拓扑不得包含当前不支持的字段；
- 拓扑两端必须是不同且已定义的 Agent；
- `channel_id` 必须非空且不得重复；
- 网络 profile 字段必须能够完成规范化；
- 上传后的核心校验失败时，必须删除已持久化剧本并清理临时数据；
- 批量上传和批量解析必须按剧本逐项隔离错误。

当前上传成功只能解释为：

> 剧本通过了当前已实现的核心文件、Agent 基础字段和通信拓扑校验。

在本文所有待完成项落地前，不得把当前上传成功解释为“完整语义校验通过”。

## 2. 待完成校验项

### 2.1 Skill 实体与引用完整性

状态：**待完成**。

需要补充：

- 解析剧本中的 Skill 定义集合；
- 校验每个 `skill_ref` 指向的 Skill 文件或目录真实存在；
- 校验 Skill 标识唯一；
- 校验 Skill 定义的必填字段、字段类型和未知字段；
- 校验 Skill 包入口文件、说明文件和依赖资源的一致性；
- 拒绝跨出当前剧本目录的 Skill 路径；
- 对不存在、重复、不可读取或结构非法的 Skill 返回明确错误位置。

### 2.2 Tool 实体与引用完整性

状态：**待完成**。

需要补充：

- 解析独立 Tool 定义集合；
- 校验每个 `tool_ref` 指向的 Tool 定义真实存在；
- 校验 Tool 标识唯一；
- 校验 Tool 名称、调用协议、输入 schema 和必要配置；
- 校验 Tool 定义中的未知字段和错误类型；
- 校验场景 Tool 与全局 MCP Tool 的引用边界；
- 对不存在、重复或结构非法的 Tool 返回明确错误位置。

### 2.3 任务结构与引用关系

状态：**待完成**。

需要补充：

- 校验每个 Agent 的任务集合类型；
- 校验任务标识、名称、目标和输入字段；
- 校验任务标识在剧本内唯一；
- 校验任务引用的 Agent、Skill、Tool 和目标对象真实存在；
- 校验任务依赖不存在自引用和无效引用；
- 校验任务列表中的空值、重复值和错误类型；
- 对任务结构错误返回 Agent、任务和字段位置。

### 2.4 Agent 集合完整性

状态：**待完成**。

需要补充：

- 剧本必须至少包含一个 Agent；
- Agent 标识必须非空并满足统一命名规则；
- Agent 标识大小写规范化后不得重复；
- `roles` 和 `container_instances` 必须双向匹配；
- 不得存在没有角色定义的容器实例；
- 不得存在没有运行实例定义的角色；
- Agent 名称、身份、目标和任务字段必须满足必填及类型约束；
- Agent 后端能力必须与其引用的 Skill 和 Tool 兼容。

### 2.5 核心 JSON 严格 schema

状态：**待完成**。

需要补充：

- 为三个核心 JSON 文件建立独立、版本化的严格 schema；
- 校验根对象类型；
- 校验必填顶层字段；
- 拒绝未知顶层字段；
- 校验嵌套对象、数组和标量字段类型；
- 校验枚举值、数值范围和字符串约束；
- schema 版本变更时同步更新解析器、示例剧本、测试和设计文档；
- 不保留旧结构的隐式兼容映射。

### 2.6 跨文件一致性

状态：**待完成**。

需要补充：

- `meta_and_roles.json` 中的角色与 `instances_and_skills.json` 中的实例完全对应；
- Agent、Skill、Tool、任务和拓扑引用形成闭合引用集合；
- 删除或改名后的对象不得留下悬空引用；
- 同一个业务标识不得在不同实体类型中产生歧义；
- 拓扑覆盖规则和是否允许孤立 Agent 必须形成明确合同；
- 校验结果必须包含所有发现的问题，而不是只返回第一个异常。

## 4. 目标校验结果

完整校验完成后，单个剧本校验结果至少应包含：

```text
scene_key
validation_status
schema_version
validated_at
error_count
warning_count
issues[]
```

每个 `issue` 至少包含：

```text
code
severity
file
json_path
entity_type
entity_id
message
```

目标状态建议固定为：

```text
core_validated
fully_validated
invalid
```

在待完成项全部落地前，现有成功结果只能映射为 `core_validated`。

## 5. 验收标准

全部待完成项只有同时满足以下条件后才能改为“已完成”：

1. Skill、Tool、任务和 Agent 集合均有独立结构模型；
2. 三个核心 JSON 文件均有严格、版本化 schema；
3. 所有跨文件引用均被校验；
4. 校验能够一次返回多个可定位问题；
5. 单个上传、批量上传、单个解析和批量解析使用同一校验器；
6. 校验失败不会留下正式剧本、临时文件或文件注册表残留；
7. 单元测试覆盖每类非法结构和引用错误；
8. API 测试覆盖批量逐项隔离；
9. 示例剧本全部通过完整校验；
10. `docs/剧本管理设计.md`、数据模型、类操作设计和开发文档同步更新。

## 6. 禁止回退规则

- 不得在 `SceneManager`、API 或仿真编排中分别实现不同校验规则；
- 单剧本校验必须由 `SceneStorage` 调用统一剧本校验器；
- `SceneManager` 只负责编排批量校验、逐项隔离和结果汇总；
- 文件存在性和安全读取必须继续通过 `FileManager`；
- 不得仅检查引用字符串非空就宣称引用完整；
- 不得在完整校验落地前把当前上传结果标记为 `fully_validated`。
