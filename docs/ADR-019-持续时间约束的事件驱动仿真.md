# ADR-019：仿真采用持续时间约束的事件驱动调度

## 状态

已接受。本文决策覆盖并替代现有设计文档中以固定执行计数、停滞计数或调度循环编号控制仿真的目标设计描述。

## 决定

仿真调度固定采用 `event_driven` 模式：事件到达且目标 Agent 可用时立即调度，不以固定执行批次划分仿真进程。

每次仿真由 `SimulationRuntimeConfig` 独立配置：

- `duration_seconds`：最大墙钟持续时间；
- `agent_timeout_seconds`：单次 Agent 事件处理超时；
- `idle_timeout_seconds`：事件队列为空且无执行中任务时的空闲终止时间；
- `network_mode`；
- `seed`；
- `resource_limits`：CPU、内存、进程数和 Agent 并行度限制。

仿真运行以 `SimulationEvent` 为调度单位。初始任务、Agent 消息、后续任务和系统触发都必须转换为事件并进入按 `simulation_id` 隔离的事件队列。

## 调度约束

- 事件队列按可执行时间和稳定序号排序；
- 同一逻辑 Agent 默认单任务串行；
- 不同 Agent 可在资源限制范围内并行；
- 调度器在无事件时阻塞等待，不进行固定周期空转；
- 每次事件入队、派发和完成都更新仿真实例活动时间；
- 达到持续时间上限后必须停止继续派发事件；
- 取消仿真时，待处理事件进入 `cancelled`，执行中任务进入受控停止流程。

## 终止条件

允许的主要终止原因包括：

- `duration_elapsed`；
- `user_stopped`；
- `idle_completed`；
- `tasks_completed`；
- `all_agents_failed`；
- `resource_limit_exceeded`；
- `capture_incomplete`；
- `runtime_exception`；
- 启动阶段资源分配、网络配置或抓包失败。

## 禁止回退

目标模型、API、Agent 上下文、Dashboard、日志和实验清单不得重新引入以下调度合同：

- `max_rounds`；
- `stalemate_rounds`；
- `current_round`；
- `current_turn`；
- `current_max_rounds`；
- `round`；
- `tick`；
- `total_rounds`；
- `event_driven_rounds`；
- 以固定次数循环作为仿真持续时间或结束条件。

不得从剧本元数据隐式读取旧调度字段并覆盖本次仿真的运行配置。

## 对 ADR-014 的修订

ADR-014 中“运行配置归属于仿真实例”的所有权决定继续有效，但其旧字段列表由本文定义的 `SimulationRuntimeConfig` 替代。ADR-014 中关于固定执行计数、停滞计数和当前执行批次的描述不再有效。

## 当前实现差距

当前代码仍包含固定次数循环、全局执行进度状态和旧上下文字段。实现迁移时必须同步修改：

- 仿真创建与启动请求；
- `SimulationRun` 和运行配置模型；
- Agent 执行调度器；
- Dashboard 状态展示；
- application/system 日志生产者与消费者；
- experiment manifest 版本与质量审计；
- 场景中遗留调度字段的读取逻辑；
- 单元测试和端到端测试。

项目处于初期开发阶段，迁移时直接删除旧合同，不保留兼容映射或双轨调度器。
