# 通信与网络仿真设计

统一通信入口、A2A 协议合同、禁止广播和防回退规则以 [ADR-024-CommManager统一A2A通信与禁止广播.md](ADR-024-CommManager统一A2A通信与禁止广播.md) 为权威依据；Agent 执行任务、持久化状态机和 Push 回调以 [ADR-025-Agent任务下发与A2A回调.md](ADR-025-Agent任务下发与A2A回调.md) 为权威依据。

## 1. 当前通信模式

当前唯一有效的数据面模式是 `a2a`。Agent 使用统一的 `CommManager`，按照 A2A 1.0 HTTP+JSON 绑定向目标 Agent 发送点对点请求。

```text
Agent A MCP send_message
  -> CommManager 权限检查与 Agent Card 发现
  -> POST http://<container-B>:8000/a2a/v1/message:send
  -> Agent B inbox
  -> 下一轮 /run 消费消息
```

执行任务使用相同的点对点链路，但由 `delegate_task` 设置 `operation=agent.task.assign`。目标返回 `SUBMITTED` 后异步执行，并向来源 Agent 或 `srv` 发送带 token 的 A2A Push 状态与 Artifact 回调。多个任务目标仍必须逐个发送，不允许广播或并发 fan-out。

`services/message_bus.py` 已废弃，只保留兼容提示；`/relay` 返回 410。不得在新代码或文档中把它描述成当前通信组件。

## 2. Agent 目录与通信矩阵

`srv` 在 launch 后生成：

- `agent_directory`: `agent_id -> Agent 基础 URL`，由 `CommManager` 获取 Agent Card
- `comm_matrix`: `agent_id -> allowed target agent IDs`

两者在 Agent 执行前通过 `/communication/configure` 注入，并随每次 `/run` 刷新到 Agent 容器和 MCP runtime 的 `CommManager`。

规则：

- ID 全部标准化为小写；
- 只接受精确 `agent_id`，不做模糊名称或角色匹配；
- 存在通信矩阵时，目标不在 allowlist 即返回失败且不发请求；
- 当前拓扑边被展开为双向权限，因此矩阵是对称的；
- 不提供广播能力；多个明确目标必须按输入顺序逐个调用 `send_message`，并分别记录结果。

正式仿真必须由 `srv` 注入从拓扑生成的矩阵。发送端和接收端都执行同一权限检查，不允许模糊目标、通配符目标或自发自收。

## 3. 拓扑的双重用途

`network_topology.json` 中的每条边同时用于：

1. 生成逻辑通信权限；
2. 生成两端的出站网络仿真 profile。

支持的链路参数：

| 字段 | 范围 | 含义 |
|---|---:|---|
| `delay_ms` | 0–60000 | 出站附加时延 |
| `jitter_ms` | 0–60000 | 出站时延抖动 |
| `loss_pct` | 0–100 | 出站丢包百分比 |
| `rate_mbit` | 0–1000000 | HTB 速率；0 表示不限制 |

没有任何非零网络参数的边只参与通信权限，不创建 `tc` profile。

## 4. Linux `tc` 实现

网络仿真在每个 Agent 容器的 `eth0` 上配置：

- 根 qdisc 使用 HTB；
- 每个目标 Agent 建立一个 class；
- 时延、抖动和丢包使用 `netem`；
- 目标匹配使用对端容器 IPv4 `/32`；
- 每个 profile 只作用于当前容器发往该目标的出站包。

为了模拟对称链路，`srv` 会为拓扑边两端分别下发 profile。任一请求 profile 配置失败时，控制面清理已经设置的 qdisc，并终止实验启动。

## 5. 真实抓包

抓包在每个 Agent 容器网络命名空间中执行：

```text
tcpdump -i any -nn -s 0 -U -w <agent_id>.pcap [BPF filter]
```

默认排除发往 `srv` 的控制和日志流量，以保留 Agent-to-Agent、LLM、MCP、DNS 等运行时流量。设置 `AGENT_CAPTURE_INCLUDE_CONTROL_PLANE=1` 才包含控制面流量。

每个 PCAP 同时写入 sidecar manifest，记录：

- 逻辑 Agent ID；
- 运行容器名、容器 ID、IP 和后端；
- session/trace；
- interface、BPF filter 和网络 profile；
- 开始/停止时间、文件大小、状态和 SHA-256。

超过 `PCAP_MAX_BYTES` 时 watchdog 终止抓包并将实验标记为错误。

## 6. 网络记录来源

`network.jsonl` 只能从 PCAP 中的真实 packet record 产生。HTTP 中间件、Tool 调用或 Agent 消息事件不得直接伪造 `network.jsonl`。

结构化查询使用 `tcpdump -tt -nn -r` 解码，并保留：

- 源/目的 IP 与端口；
- IP 版本；
- TCP/UDP/ICMP/IP 协议；
- TCP flags；
- 相对当前 Agent 的 inbound/outbound/observed；
- Agent peer 与 non-peer 分类；
- IP payload 长度；
- 原始解码行与 PCAP 路径。

“non-peer”包括 LLM、MCP、互联网、DNS 和容器基础设施；在没有权威网络资产清单时不得一概标记为 external。

## 7. 跨层关联

`/api/packets/correlate` 使用应用事件结束时间、事件持续时间和可配置时间窗口匹配包。它是 `temporal_window_inference`：

- 可用于发现某个应用事件附近的协议、方向和字节证据；
- 不能证明某个 packet 一定由某个应用事件因果触发；
- 结果中必须保留这一 caveat。

## 8. 设计约束

- 不恢复中心 relay 数据面，除非有明确的新架构决策并同步迁移所有调用方。
- 通信权限检查必须发生在实际发送前。
- 网络仿真失败不得静默降级。
- 网络日志必须可回溯到真实 PCAP。
- 不用应用层字段冒充网络层字段。
