# ADR-021：抓包生命周期统一经过 CaptureManager

## 状态

已接受并实施第一版。

## 决定

AgentNetwork 的抓包生命周期、目标状态聚合、失败回滚、健康检查、PCAP 最终投影和抓包资源查询统一经过 `agent_network.capture_management.CaptureManager`。

每个 Agent 容器内只能存在一个 `CaptureRuntime` 实例负责当前网络命名空间中的 `tcpdump` 进程。

## 权威事实

- 原始 PCAP 是网络层和传输层事实的唯一权威来源；
- `network.jsonl` 只能由真实 PCAP 投影生成；
- 应用事件、模型输出、Tool 返回或估算流量不得生成 packet 记录；
- 同一 Agent-to-Agent 数据包可能在两端 PCAP 中各观察一次，统计必须明确采用 `per_agent_observations` 口径。

## 统一职责

`CaptureManager` 负责：

- 创建抓包会话；
- 并发启动所有 Agent 目标；
- 部分启动失败时回滚；
- 聚合健康状态；
- 停止所有目标；
- 驱动最终 PCAP 投影；
- 提供分析、审计、下载和 bundle 的统一门面。

`CaptureRuntime` 负责：

- 启动和停止 `tcpdump`；
- BPF 过滤；
- 最大 PCAP 字节限制；
- 进程健康与强制终止；
- capture manifest；
- PCAP 文件刷新和哈希。

`FileManager` 继续负责物理文件生命周期。抓包模块不得实现第二套路径、压缩、可见性、哈希或注册表机制。

## 状态合同

抓包会话和抓包目标采用：

```text
CREATED
STARTING
RUNNING
STOPPING
STOPPED
FAILED
INCOMPLETE
```

仿真事件调度只能在全部预期目标进入 `RUNNING` 后启动。

任一目标异常退出、超限、失联、文件缺失、哈希失败或投影失败时，会话必须进入 `INCOMPLETE` 或 `FAILED`，不得报告为完整实验。

## 投影合同

停止后采用 `projection_mode=finalize`：

1. 刷新 PCAP 大小和 SHA-256；
2. 计算源指纹；
3. 解码 PCAP；
4. 写入 `network.v4`；
5. 保存投影结果；
6. 相同源指纹不得重复写入。

幂等键包含：

```text
capture_id + observer_agent_id + pcap_sha256 + packet_index
```

投影健康检查和抓包生命周期不得依赖固定仿真轮次、round 或 tick。

## API 决定

外部抓包入口统一为 `/api/captures`。`/api/packets` 不再挂载到活动控制面服务。

## 禁止回退

- 不得恢复 `PacketRecorder` 作为真实网络数据来源；
- 不得在仿真编排中重新实现逐 Agent 抓包扇出和状态聚合；
- 不得让 Agent Runtime 之外的组件拥有其网络命名空间中的 `tcpdump`；
- 不得把应用日志、HTTP 中间件日志或合成流量字段写入 `network.jsonl`；
- 不得在抓包模块中绕过 `FileManager` 直接提供下载或维护可见性；
- 不得让抓包健康检查依赖仿真轮次；
- 不得在 PCAP 源指纹不变时重复投影网络日志；
- 不得向外部 API 返回物理文件路径。

## 当前迁移说明

`real_packet_store.py` 目前仍承载 PCAP 解码和分析实现，`experiment_manifest.py` 仍承载实验来源、质量审计和 bundle 语义。它们已经通过 `CaptureManager` 统一暴露，不再作为活动外部抓包入口。

后续拆分内部实现时必须保持本 ADR 的统一入口、状态合同和证据边界。
