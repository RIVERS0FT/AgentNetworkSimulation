# 统一日志管理设计

> 状态：已实现第一版。本文是日志批量管理、会话可见性、文件可见性、解析、下载和删除的权威设计。

## 1. 目标

`LogManager` 与 `SceneManager` 保持同一级别的批量业务管理能力。日志记录、schema 规范化、实时查询和持久化仍由同一个 `LogManager` 提供，不新增第二套日志管理器。

统一日志管理支持：

- `application.jsonl`、`network.jsonl`、`system.jsonl` 的规范化写入；
- 会话日志读取和 schema 解析；
- 跨会话、跨日志类型的批量下载；
- 跨会话、跨日志类型的批量删除；
- 批量解析与逐项错误隔离；
- 批量修改单个日志文件可见性；
- 修改整个日志会话可见性；
- 批量修改会话可见性；
- 批量删除非活动日志会话；
- 统一批量结果和失败原因。

## 2. 架构

```text
Log API / Agent Runtime / Capture Projection
                    |
                    v
                LogManager
      +-------------+--------------+
      |             |              |
 schema规范化   实时内存索引    批量业务编排
      |                            |
      +-------------+--------------+
                    |
                    v
                FileManager
                    |
                    v
 data/logs/<session_id>/{application,network,system}.jsonl
```

`LogManager` 是日志领域唯一入口。`FileManager` 只负责文件生命周期，不理解日志 schema、事件类型、活动会话或批量业务语义。

## 3. 批量结果模型

### 3.1 `LogBatchItemResult`

| 字段 | 描述 |
|---|---|
| `operation` | `download`、`delete`、`parse`、`hide`、`show`、`delete_session` 等 |
| `session_id` | 日志会话标识 |
| `log_type` | `application`、`network`、`system`；会话级操作使用 `*` |
| `success` | 当前项目是否成功 |
| `status` | 项目状态 |
| `resource_id` | 受管日志资源标识 |
| `error_code` | 稳定错误代码 |
| `error` | 失败原因 |
| `details` | 解析计数、文件名、大小、哈希等操作明细 |

### 3.2 `LogBatchResult`

统一返回：

```text
operation
batch_id
total
succeeded
failed
items[]
archive_resource_id
archive_name
```

一个项目失败不得中断其他项目。重复引用作为当前项目失败返回，不静默去重。

## 4. 可见性模型

日志可见性包含两级：

```text
会话目录可见性
        AND
单个日志文件可见性
        =
有效可见性
```

规则：

1. 隐藏会话时，同时隐藏会话目录和当前全部日志文件；
2. 显示会话时，同时显示会话目录和当前全部日志文件；
3. 会话隐藏期间新创建的日志文件必须继承隐藏状态；
4. 单文件显示不能绕过隐藏的父会话；
5. 默认列表、解析和下载使用有效可见性；
6. 内部审计可显式设置 `allow_hidden=true` 读取隐藏日志；
7. 隐藏不等于删除，写入仍可继续；
8. `include_hidden=true` 的列表必须同时返回会话可见性、文件可见性和有效可见性。

## 5. 批量能力

### 5.1 批量下载

输入多个 `session_id + log_type`。每个可读资源进入同一个 ZIP，归档路径为：

```text
<session_id>/<log_filename>
```

归档包含 `LOG_BATCH_MANIFEST.json`，记录批次标识和逐项结果。隐藏会话或隐藏日志默认不进入归档。

### 5.2 批量解析

逐行解析 JSONL，并执行对应 schema 校验。每个文件返回：

```text
valid_records
invalid_records
errors[{line,error}]
```

单行失败不得终止当前文件后续记录，也不得终止其他文件。错误明细数量必须有限制。

### 5.3 批量删除

支持：

- 删除多个指定日志文件；
- 删除多个完整日志会话。

活动日志会话受保护，不得通过批量文件删除或批量会话删除移除。非活动项目失败不影响其他项目。

### 5.4 批量可见性

支持：

- 多个指定日志文件批量显示或隐藏；
- 多个日志会话批量显示或隐藏。

批量操作使用统一逐项结果，不因部分资源不存在而整体回滚。

## 6. API

```text
POST /api/logs/batch/download
GET  /api/logs/batch/download/{resource_id}
POST /api/logs/batch/delete
POST /api/logs/batch/parse
POST /api/logs/batch/visibility

POST /api/logs/sessions/{session_id}/hide
POST /api/logs/sessions/{session_id}/show
POST /api/logs/sessions/batch/visibility
POST /api/logs/sessions/batch/delete
```

原有单文件下载、显示、隐藏、删除和单会话删除接口继续由同一个 `LogManager` 提供。

## 7. 实现入口

```text
agent_network/log_manager.py
    日志 schema、规范化、写入、内存索引和基础管理接口

agent_network/file_management/log_integration.py
    将日志物理操作切换到 FileManager

agent_network/log_batch.py
    为当前 UnifiedLogManager 安装批量业务和会话可见性能力

agent_network/api/logs.py
    原有日志查询、写入和单资源管理 API

agent_network/api/log_batches.py
    批量日志和会话可见性 API
```

## 8. 设计约束

1. 不得新增第二个日志管理单例；
2. 所有批量操作必须由当前 `LogManager` 执行；
3. API 不得直接调用 `FileManager` 操作日志资源；
4. 日志物理写入、读取、归档、可见性和删除必须委托 `FileManager`；
5. `LogManager` 继续负责 schema、日志类型和活动会话语义；
6. 活动会话不得通过批量删除移除；
7. 会话隐藏必须成为所有子日志的父级可见性门控；
8. 隐藏会话期间的新日志文件不得恢复为可见；
9. 批量下载必须生成受管归档并经过完整性校验；
10. 批量解析不得写回或修改原始 JSONL；
11. 批量结果必须逐项返回成功、失败和失败原因；
12. 不得恢复 `.log_visibility.json` 作为活动可见性机制。
