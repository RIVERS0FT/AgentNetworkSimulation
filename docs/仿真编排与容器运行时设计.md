# 仿真编排与容器运行时设计

## 1. 设计目标

编排层负责把静态场景转换为一次可审计的多 Agent 实验。逻辑 Agent 与容器槽位分离：逻辑身份来自场景，容器只是可复用的后端执行资源。

## 2. 状态模型

控制面维护两类对象：

- `Agent`：逻辑身份、角色、目标、后端、Skill/Tool allowlist 和展示状态；
- `ContainerAgent`：本次分配的容器 ID、容器名、IP、镜像、URL 和运行状态。

一个容器在一次分配中只承载一个逻辑 Agent。新实验分配前执行 `/reset`，清除 inbox、事件队列、抓包进程和网络仿真配置。

### 2.1 当前并发限制

当前实现同一时间最多允许一个仿真处于启动、运行或停止阶段，并发上限等价于：

```text
max_concurrent_simulations = 1
```

当前代码使用单一全局状态表达仿真：

- `simulation_active`：是否存在运行中的仿真；
- `current_scene_name`：当前加载或运行的剧本；
- `current_turn`：当前轮次；
- `current_max_rounds`：当前最大轮数。

当前约束如下：

1. 新仿真开始前必须确认运行数量小于并发上限；
2. 当前上限为 `1`，所以仿真运行期间不得启动第二个仿真；
3. 停止、失败或正常结束后必须释放运行状态，才能启动下一次仿真；
4. 同一轮中的多个 Agent 可以并发执行，但这不等于多个仿真实例并发运行。

### 2.2 后续多仿真扩展边界

“当前只运行一个仿真”是可调整的运行策略，不是永久领域约束。未来提高并发上限时，应保持现有剧本管理和运行接口语义，替换控制面内部状态实现。

扩展时应完成：

1. 使用 `simulation_id` 标识每个仿真实例；
2. 将单一全局状态替换为按 `simulation_id` 索引的仿真运行注册表；
3. 每个仿真实例独立保存剧本、轮次、Agent 注册表、容器分配、通信矩阵和终止配置；
4. 每个仿真实例独立拥有日志 session、PCAP、manifest、网络 profile 和清理流程；
5. 容器槽位必须由资源调度器分配，禁止同一运行容器同时承载两个逻辑 Agent；
6. 停止、状态查询、日志查询和结果下载接口必须能够指定 `simulation_id`；
7. 剧本占用查询返回运行引用集合，而不是单一当前仿真字段；
8. 并发上限由配置或容量策略控制，默认仍可设为 `1`。

因此，外部设计不得把 `simulation_active`、`current_scene_name` 等当前实现字段固化为长期接口合同。

## 3. 场景加载契约

控制面只读取：

```text
scenes/<scene>/
  meta_and_roles.json
  instances_and_skills.json
  network_topology.json
```

加载器执行严格校验：

- scene 角色对应逻辑 `agent_id`；
- `skill_refs` 必须是非空字符串并去重；
- 场景后端仅允许 `openclaw` 或 `claude-code`；
- 拓扑根字段必须是 `topology` 数组；
- 每条边只能包含 `endpoint_a`、`endpoint_b`、`channel_id`、`delay_ms`、`jitter_ms`、`loss_pct`、`rate_mbit`；
- 端点必须存在、不同且标准化为小写；
- `channel_id` 必须非空且在场景内唯一；
- 网络参数必须在 `network_emulation.normalize_profile` 的范围内。

场景写法仍以 `scenes/` 下文档为准；本文件只描述控制面的消费契约。

## 4. 容器分配

`ContainerRuntime` 按后端选择镜像：

| 后端 | 镜像 | 默认容器前缀 |
|---|---|---|
| Claude Code | `agentnetwork-ag-c1` | `ag-c` |
| OpenCLAW | `agentnetwork-ag-o1` | `ag-o` |

分配顺序：

1. 复用同后端、尚未被本次实验占用的运行容器；
2. 若无可用槽位且 Docker SDK 可用，则动态创建容器；
3. 动态容器加入 `an` 网络，挂载代码、只读 scenes 和可写 PCAP 目录，并授予 `NET_RAW`、`NET_ADMIN`；
4. 若无法分配，记录 assignment error，并在实验 manifest 中以 `assignment_failed` 结束。

容器名不是逻辑 Agent ID。所有业务日志必须使用 `/run` 中传入的顶层逻辑 `agent_id`。

未来支持多仿真时，容器分配记录必须同时关联 `simulation_id` 和逻辑 `agent_id`。

## 5. Launch 前置阶段

启动实验时依次执行：

1. 检查当前运行数量小于并发上限；
2. 创建本次仿真的运行标识；
3. 重置并分配本次仿真使用的容器；
4. 构造 Agent 目录和通信矩阵；
5. 启动日志 session；
6. 创建 `agent-traffic-experiment.v1` manifest；
7. 配置网络 profile；
8. 启动所有 Agent 的 PCAP；
9. 检查上述步骤是否完整。

当前实现只有一个全局运行槽位，因此第 1 步等价于检查平台当前没有运行中的仿真。

已请求的网络仿真无法配置，或任一 Agent 抓包无法启动时，实验不得继续运行。

## 6. 轮次调度

当前调度模式为 `event_driven_rounds`：

- 第一轮向各 Agent 投递场景初始任务；
- 后续轮次不重复投递初始任务；
- 有任务或 inbox 消息的 Agent 才调用 `/run`；
- 无任务且 inbox 为空的 Agent 返回 `skipped`；
- 同一轮中的 Agent 使用线程池并发执行，最大并发数为 10；
- 每轮后检查抓包健康状态。

Agent 并发属于单次仿真内部并发。未来仿真实例并发时，每个实例仍应保持独立的轮次调度上下文。

`AgentContext` 是后端无关的输入合同，`AgentRunResult` 是后端无关的输出合同。后端特有 SDK 对象不得泄漏到控制面接口。

## 7. 终止条件

实验可能因以下原因停止：

- 达到 `max_rounds`：`hard_limit`；
- 用户请求停止：`user_stopped`；
- 连续达到 `stalemate_rounds` 个无有效输出轮次；
- 全部 Agent 执行失败：`all_agents_failed`；
- 抓包中途停止：`capture_incomplete`；
- 运行时异常：`runtime_exception`；
- 启动阶段分配、网络仿真或抓包失败。

“有效输出”由 application events、tool events 或非空 final message 计算。

## 8. 收尾与实验状态

`finally` 阶段必须执行：

1. 停止本次仿真的所有 Agent 抓包；
2. 清理本次仿真的所有 `tc` 配置；
3. 释放本次仿真占用的容器槽位；
4. 完成 experiment manifest；
5. 执行 `audit_session`；
6. 从运行注册表中释放本次仿真状态。

当前实现中，第 6 步表现为清除全局运行中标志。未来多仿真模式下，只能移除对应 `simulation_id`，不得影响其他仿真。

只有没有运行时错误、没有抓包不完整且抓包停止成功时，实验状态才为 `complete`。

## 9. 设计约束

- 控制面不得在本地替代 Agent 后端执行推理。
- 不得把容器静态环境中的 `AGENT_ID` 当作最终业务 Agent ID。
- 初始任务不得在每轮重复注入。
- 抓包和网络仿真是实验完整性的一部分，不是可忽略的附加功能。
- 当前并发仿真上限为 `1`，但不得把单例全局状态固化为长期外部接口。
- 多仿真扩展必须保证运行状态、容器、网络配置、日志和实验产物按 `simulation_id` 隔离。
- 新增终止条件或调度语义时，必须同步更新 manifest 的 scheduler 信息与本文件。
