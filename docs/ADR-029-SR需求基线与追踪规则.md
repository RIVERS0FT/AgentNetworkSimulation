# ADR-029：SR 需求基线与追踪规则

> 状态：已接受

## 背景

项目已经分别形成剧本管理、日志管理、仿真编排、实时 Dashboard 等设计文档，但不同文档形成时间不同，部分详细合同与最新产品需求不一致。例如旧剧本管理设计曾把列表固定为“无查询条件、只返回名称”，与当前 `SR-SCENE-05` 要求的“查询并展示名称、状态和基本信息”冲突。

项目处于早期开发阶段，不保留无需求的兼容设计。需要一个统一、可追踪、不可被实现现状反向削弱的 SR 基线。

## 决定

1. [系统需求规格.md](系统需求规格.md) 是所有 SR 的权威基线。
2. 详细设计必须通过 SR-ID 映射到该基线，不得自行改写需求含义。
3. 实现状态使用“已实现第一版、部分实现、设计阶段”等标签表达，不得把未实现部分从需求文档删除。
4. 详细设计与 SR 基线冲突时，以 SR 基线为准，并在同一变更中修正文档冲突。
5. 修改 SR 时，必须同步更新：
   - 系统需求规格；
   - 相关领域设计；
   - API 与数据模型合同；
   - 验收或回归测试计划。
6. SR-ID 不复用。废弃需求保留原 ID、废弃原因和替代需求。
7. 产品术语和技术模型可以不同，但映射必须明确。例如“预览剧本”可以由只读 `SceneSummary` 实现，不要求创建名为 `ScenePreview` 的类型。

## 本次固定的防回退合同

### 剧本管理

- `SR-SCENE-04` 的产品能力名称为“预览剧本”；预览必须发生在仿真开始前，且无运行副作用。
- `SR-SCENE-05` 要求查询已保存剧本并展示名称、状态和基本信息。
- 旧的“列表无查询参数、只展示名称、禁止状态和基本信息”合同不再是目标设计。
- 是否新建 `ScenePreview` 类型由详细设计决定；不得以不创建该类型为由取消预览能力。

### 日志管理

- `application.jsonl` 是 Agent 行为与传统应用层日志。
- `network.jsonl` 是由真实 PCAP 投影得到的原始网络层与传输层日志。
- `system.jsonl` 是附加的平台调试日志，不能替代前两者。
- 隐藏是可见性变更，不等于删除。

### 仿真模拟

- 仿真持续时间和资源限制归属于仿真实例，不写入静态剧本定义。
- 手动停止、最大持续时间终止和全局任务完成终止均为必须保持的生命周期能力。

### 实时展示

- 通信流只由真实消息事件或可验证网络事实驱动，不生成虚假动画。
- Token 图展示当前仿真的累计消耗，不得使用随机或推测数据。
- 剧本自定义面板采用声明式 schema、白名单组件和受控数据绑定，不执行任意上传脚本。

## 影响

- 现有剧本管理查询接口和列表模型需要后续迁移，以满足 `SR-SCENE-05`。
- 现有实时展示能力需要形成统一的快照、增量、重连和安全面板合同。
- 后续代码变更必须先检查 SR 追踪，不允许仅依据旧实现或单个旧文档恢复被本 ADR 取代的约束。

## 禁止回退

- 不得恢复“只返回剧本名称且禁止查询”的永久合同。
- 不得把详情接口存在当作取消“预览剧本”产品能力的理由。
- 不得把应用日志事件转换成伪造的网络层记录。
- 不得把 Dashboard 动画当作网络事实来源。
- 不得允许剧本包携带并执行任意浏览器脚本。
