# AgentNetwork 总体设计

> 当前技术栈：Python、FastAPI、Docker / Docker Compose、OpenCLAW、Claude Agent SDK、MCP、Linux `tc`、`tcpdump`、原生 Web Dashboard。

## 1. 系统目标

AgentNetwork 用于在可重复实验条件下运行多个逻辑 Agent，观察它们的任务执行、工具使用、消息交互和真实网络流量。平台把“逻辑 Agent”与“后端运行容器”分离，并同时保存应用行为证据、原始网络证据和实验来源信息。

当前系统的关键目标是：

- 使用场景配置定义 Agent 身份、后端、Skill/Tool allowlist 和通信拓扑；
- 由 `srv` 完成场景加载、仿真创建、事件调度、容器分配、抓包控制和结果汇总；
- 在 Agent 容器内运行真实 OpenCLAW 或 Claude Code 后端；
- 使用 Agent-to-Agent HTTP 直连通信，并按拓扑强制限制可达目标；
- 使用 Linux `tc` 对指定对端施加时延、抖动、丢包和带宽限制；
- 使用容器网络命名空间内的 `tcpdump` 采集真实数据包；
- 将 Agent 行为和传统应用层事件写入 `application.jsonl`；
- 将原始网络层和传输层数据包写入 `network.jsonl`，并保留原始 PCAP；
- 通过实验 manifest、哈希和质量审计保证结果可追溯。

目标仿真模型采用持续时间约束的事件驱动调度。固定执行计数、停滞计数和调度循环编号不属于目标设计合同。

## 2. 总体架构

```text
Browser / API Client
        |
        v
+---------------------------+
| srv: FastAPI control plane|
| - scene setup / launch    |
| - simulation registry     |
| - event dispatcher        |
| - container scheduling    |
| - log / packet APIs       |
| - dashboard / WebSocket   |
+-------------+-------------+
              |
              | Docker API + HTTP /run
              v
+---------------------------+       +---------------------------+
| reusable Agent container A| <---> | reusable Agent container B|
| - services/agent_server.py| HTTP  | - services/agent_server.py|
| - backend adapter         | /msg  | - backend adapter         |
| - MCP tools               |       | - MCP tools               |
| - tc network emulation    |       | - tc network emulation    |
| - tcpdump full capture    |       | - tcpdump full capture    |
+-------------+-------------+       +-------------+-------------+
              |                                   |
              +------------ shared volumes -------+
                       data/logs, data/pcap
```

系统没有运行中的中心消息总线数据面。`services/message_bus.py` 仅是废弃兼容桩，`/relay` 返回 HTTP 410；新的设计不得把 Agent 通信重新接回该服务。

## 3. 组件职责

| 组件 | 当前或目标职责 | 明确不负责 |
|---|---|---|
| `srv` | 场景解析、仿真实例管理、事件调度、容器分配、抓包和网络仿真控制、日志与实验汇总 | 不执行 Agent 推理；不读取 Skill 正文；不执行场景 Tool |
| `SimulationRun` | 持有一次仿真的运行配置、状态、截止时间、最近活动时间和事件统计 | 不持有静态剧本资源内容 |
| 事件调度器 | 管理按 `simulation_id` 隔离的事件队列，事件到达即派发，并执行持续时间、空闲时间和停止检查 | 不使用固定执行批次或周期性空转作为调度合同 |
| `ContainerRuntime` | 复用或创建后端容器、分配逻辑 Agent、应用资源限制、并发调用 `/run` | 不理解 Skill 内容；不代理 Agent 间消息 |
| Agent 容器 | 承载一个当前分配的逻辑 Agent，维护 inbox，运行后端、MCP、抓包与 `tc` | 不拥有永久业务身份；重分配前必须 reset |
| Backend Adapter | 把统一 `AgentContext` 映射到 OpenCLAW/Claude Code，并返回 `AgentRunResult` | 不扩大 Skill/Tool 权限；不伪造网络日志 |
| CommManager | 依据 `agent_directory`、Agent Card 与 `comm_matrix` 使用 A2A 1.0 点对点通信 | 不广播；不进行模糊名称匹配；不绕过通信矩阵 |
| TaskManager / CallbackDispatcher | 持久化 Agent 执行任务状态，发送和接收带认证的 A2A Push 回调 | 不执行 Agent 推理；不把回调失败等同业务失败 |
| MCP server | 注册系统消息工具和 allowlist 内的场景原子 Tool | 不把 Skill 名称注册成 Tool |
| Skill source | 向后端提供 allowlist 内 Skill 源文件 | 不执行 Skill；不允许路径逃逸 |
| LogManager | 校验并持久化 application/network/system 三类日志 | 不从应用请求推断虚假数据包 |
| PCAP 子系统 | 在 Agent 网络命名空间真实抓包、解析、分析、打包 | 不声称时间窗口相关性等于协议级因果关系 |

## 4. 配置与运行数据流

### 4.1 Setup

目标仿真创建请求：

1. 读取并校验静态剧本；
2. 接收 `duration_seconds`、Agent 执行超时、空闲超时、随机种子、网络模式和资源限制；
3. 校验当前并发容量和资源可用性；
4. 创建 `simulation_id`、`SimulationRun` 和独立事件队列；
5. 把剧本中的初始任务转换为待处理事件。

剧本解析不得读取或修改仿真运行配置，也不得改变当前运行状态。

### 4.2 Launch

仿真启动流程：

1. 为每个逻辑 Agent 分配与后端匹配的可复用容器，池不足时可动态创建；
2. 对容器执行 reset，并应用本次仿真的 CPU、内存和进程数限制；
3. 构建 `agent_id -> Agent URL` 目录和对称通信矩阵，并在执行前配置各容器的 `CommManager`；
4. 从拓扑生成每个 Agent 的出站 `tc` profile；
5. 创建实验 manifest，记录场景哈希、种子、持续时间、资源限制、容器镜像身份和脱敏 LLM 配置；
6. 启动每个 Agent 的真实 PCAP；
7. 启动事件调度器；
8. 在终止后停止抓包、清理 `tc`、完成 manifest，并执行质量审计。

### 4.3 单个事件执行

事件调度器向容器 `/run` 传递统一上下文：

- 逻辑 Agent 身份、角色和目标；
- 当前事件对应的任务与 inbox 消息；
- `scene_key`、`skill_refs`、`allowed_tools`；
- `agent_directory` 与 `comm_matrix`；
- `trace_id`、随机种子和状态快照。

上下文不得携带固定执行批次编号、调度循环编号或 tick。

Agent 容器选择后端 Adapter，运行后将标准化应用事件提交到 `srv /api/logs/ingest`。Agent 间消息由 MCP 的 `send_message` 交给 `CommManager`，经 A2A `message:send` 点对点发送；多个明确目标按顺序逐个发送，不提供广播。

Agent 间执行任务由 MCP 的 `delegate_task` 下发；仿真编排也可用 `srv` 身份向一个明确 Agent 下发。目标先持久化为 `SUBMITTED`，由 `/run` 领取执行，并通过已登记 URL 和 token 回调 `WORKING`、Artifact 以及终态。Agent Server 与 MCP runtime 按容器环境 `AGENT_ID` 共享 `data/tasks/{runtime_agent_id}.db`，编排器使用 `data/tasks/orchestrator.db`。

同一逻辑 Agent 默认单任务串行，不同 Agent 可以在资源限制范围内并行执行。

## 5. 持久化与证据层

每个 session 的主要产物：

```text
data/logs/<session_id>/
  application.jsonl
  network.jsonl
  system.jsonl

data/pcap/<session_id>/
  experiment.manifest.json
  <agent_id>.pcap
  <agent_id>.manifest.json
  <session_id>.bundle.zip   # 按需生成
```

- `application.jsonl`：Agent 行为日志与传统应用层日志的统一事件流。
- `network.jsonl`：来自真实抓包的原始网络层与传输层数据包记录。
- `system.jsonl`：编排、抓包、网络配置和故障排查等系统调试日志。
- PCAP 是网络事实的原始证据；`network.jsonl` 是其结构化记录。
- 应用事件与数据包的关联目前采用时间窗口推断，只能作为证据窗口。
- experiment manifest 应记录持续时间配置、资源限制、已处理事件数、停止原因和清理结果。

## 6. 安全与权限边界

- `skill_refs` 只控制 Skill 源文件读取范围。
- `allowed_tools` 只控制可执行原子 Tool 范围。
- 通信权限由 `comm_matrix` 在 `CommManager` 发送和接收时双向强制检查。
- Skill 文件路径必须限制在当前 scene、当前 Skill 包内，拒绝绝对路径、`..` 和符号链接逃逸。
- 场景目录在 Agent 容器中只读挂载。
- 实验 manifest 中的 key、token、secret、password 字段必须脱敏。
- 控制面默认从 Agent PCAP 中排除 `/run`、抓包控制和日志回传流量；可通过显式环境变量用于调试。
- 达到持续时间或资源限制后必须停止继续派发新事件。

## 7. 当前限制与迁移差距

- 场景加载器当前只接受 `openclaw` 和 `claude-code` 后端；`direct_llm` 仅保留为显式运行时能力，不能承载需要本地 Skill 文件的 Agent。
- 通信拓扑当前按无向边构建对称矩阵。
- `tc` profile 按目标 IPv4 地址匹配出站流量；对称链路需要在两端都配置。
- 当前代码仍使用固定次数循环、全局执行进度和旧上下文字段驱动 Agent，必须迁移为按 `simulation_id` 隔离的事件调度器。
- 当前动态容器创建尚未落实每次仿真的 CPU、内存和进程数限制。
- 应用事件与网络数据的跨层关联没有协议级因果 ID，只能基于 trace、Agent 身份和时间窗口推断。

迁移时直接删除旧调度字段和兼容分支，并同步更新 API、Dashboard、日志、manifest、质量审计和测试。
