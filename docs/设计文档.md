# AgentNetwork 总体设计

> 当前技术栈：Python、FastAPI、Docker / Docker Compose、OpenCLAW、Claude Agent SDK、MCP、Linux `tc`、`tcpdump`、原生 Web Dashboard。

## 1. 系统目标

AgentNetwork 用于在可重复实验条件下运行多个逻辑 Agent，观察它们的任务执行、工具使用、消息交互和真实网络流量。平台把“逻辑 Agent”与“后端运行容器”分离，并同时保存应用行为证据、原始网络证据和实验来源信息。

当前系统的关键目标是：

- 使用场景配置定义 Agent 身份、后端、Skill/Tool allowlist 和通信拓扑；
- 由 `srv` 完成场景加载、容器分配、轮次调度、抓包控制和结果汇总；
- 在 Agent 容器内运行真实 OpenCLAW 或 Claude Code 后端；
- 使用 Agent-to-Agent HTTP 直连通信，并按拓扑强制限制可达目标；
- 使用 Linux `tc` 对指定对端施加时延、抖动、丢包和带宽限制；
- 使用容器网络命名空间内的 `tcpdump` 采集真实数据包；
- 将 Agent 行为和传统应用层事件写入 `application.jsonl`；
- 将原始网络层和传输层数据包写入 `network.jsonl`，并保留原始 PCAP；
- 通过实验 manifest、哈希和质量审计保证结果可追溯。

## 2. 总体架构

```text
Browser / API Client
        |
        v
+---------------------------+
| srv: FastAPI control plane|
| - scene setup / launch    |
| - Agent registry          |
| - container scheduling    |
| - round orchestration     |
| - log / packet APIs       |
| - dashboard / WebSocket   |
+-------------+-------------+
              |
              | Docker API + HTTP /run
              v
+---------------------------+       +---------------------------+
| reusable Agent container A| <---> | reusable Agent container B|
| - services/agent_server.py| HTTP  | - services/agent_server.py|
| - backend adapter         | /msg  | - backend adapter         |
| - MCP tools               |       | - MCP tools               |
| - tc network emulation    |       | - tc network emulation    |
| - tcpdump full capture    |       | - tcpdump full capture    |
+-------------+-------------+       +-------------+-------------+
              |                                   |
              +------------ shared volumes -------+
                       data/logs, data/pcap
```

系统没有运行中的中心消息总线数据面。`services/message_bus.py` 仅是废弃兼容桩，`/relay` 返回 HTTP 410；新的设计不得把 Agent 通信重新接回该服务。

## 3. 组件职责

| 组件 | 当前职责 | 明确不负责 |
|---|---|---|
| `srv` | 场景解析、注册表、容器分配、轮次调度、抓包和网络仿真控制、日志与实验汇总 | 不执行 Agent 推理；不读取 Skill 正文；不执行场景 Tool |
| `ContainerRuntime` | 复用或创建后端容器、分配逻辑 Agent、并发调用 `/run` | 不理解 Skill 内容；不代理 Agent 间消息 |
| Agent 容器 | 承载一个当前分配的逻辑 Agent，维护 inbox，运行后端、MCP、抓包与 `tc` | 不拥有永久业务身份；重分配前必须 reset |
| Backend Adapter | 把统一 `AgentContext` 映射到 OpenCLAW/Claude Code，并返回 `AgentRunResult` | 不扩大 Skill/Tool 权限；不伪造网络日志 |
| DirectBus | 依据 `agent_directory` 与 `comm_matrix` 直接 POST 目标 `/message` | 不进行模糊名称匹配；不绕过通信矩阵 |
| MCP server | 注册系统消息工具和 allowlist 内的场景原子 Tool | 不把 Skill 名称注册成 Tool |
| Skill source | 向后端提供 allowlist 内 Skill 源文件 | 不执行 Skill；不允许路径逃逸 |
| LogManager | 校验并持久化 application/network/system 三类日志 | 不从应用请求推断虚假数据包 |
| PCAP 子系统 | 在 Agent 网络命名空间真实抓包、解析、分析、打包 | 不声称时间窗口相关性等于协议级因果关系 |

## 4. 配置与运行数据流

### 4.1 Setup

`POST /api/simulations/setup`：

1. 读取 `meta_and_roles.json`、`instances_and_skills.json` 和 `network_topology.json`；
2. 校验后端、Skill 引用、Agent ID、拓扑端点、链路字段和 `channel_id` 唯一性；
3. 创建控制面的 `Agent` 元数据；
4. 保存待启动的 `SceneDefinition`、随机种子和终止配置。

### 4.2 Launch

`POST /api/simulations/launch`：

1. 为每个逻辑 Agent 分配与后端匹配的可复用容器，池不足时可动态创建；
2. 对容器执行 `/reset`；
3. 构建 `agent_id -> container_url` 目录和对称通信矩阵；
4. 从拓扑生成每个 Agent 的出站 `tc` profile；
5. 创建实验 manifest，记录场景哈希、种子、容器镜像身份和脱敏 LLM 配置；
6. 启动每个 Agent 的真实 PCAP；
7. 按轮并发调用 Agent `/run`；
8. 停止抓包、清理 `tc`、完成 manifest，并执行质量审计。

### 4.3 单轮 Agent 执行

`srv` 向容器 `/run` 传递统一上下文：

- 逻辑 Agent 身份、角色和目标；
- 当前任务与 inbox 消息；
- `scene_key`、`skill_refs`、`allowed_tools`；
- `agent_directory` 与 `comm_matrix`；
- `trace_id`、轮次和随机种子。

Agent 容器选择后端 Adapter，运行后将标准化应用事件提交到 `srv /api/logs/ingest`。Agent 间消息由 MCP 的 `send_message` / `broadcast` 直接发送到目标容器 `/message`。

## 5. 持久化与证据层

每个 session 的主要产物：

```text
data/logs/<session_id>/
  application.jsonl
  network.jsonl
  system.jsonl

data/pcap/<session_id>/
  experiment.manifest.json
  <agent_id>.pcap
  <agent_id>.manifest.json
  <session_id>.bundle.zip   # 按需生成
```

- `application.jsonl`：Agent 行为日志与传统应用层日志的统一事件流。
- `network.jsonl`：来自真实抓包的原始网络层与传输层数据包记录。
- `system.jsonl`：编排、抓包、网络配置和故障排查等系统调试日志。
- PCAP 是网络事实的原始证据；`network.jsonl` 是其结构化记录。
- 应用事件与数据包的关联目前采用时间窗口推断，只能作为证据窗口。

## 6. 安全与权限边界

- `skill_refs` 只控制 Skill 源文件读取范围。
- `allowed_tools` 只控制可执行原子 Tool 范围。
- 通信权限由 `comm_matrix` 在 DirectBus 发送前强制检查。
- Skill 文件路径必须限制在当前 scene、当前 Skill 包内，拒绝绝对路径、`..` 和符号链接逃逸。
- 场景目录在 Agent 容器中只读挂载。
- 实验 manifest 中的 key、token、secret、password 字段必须脱敏。
- 控制面默认从 Agent PCAP 中排除 `/run`、抓包控制和日志回传流量；可通过显式环境变量用于调试。

## 7. 当前限制

- 场景加载器当前只接受 `openclaw` 和 `claude-code` 后端；`direct_llm` 仅保留为显式运行时能力，不能承载需要本地 Skill 文件的 Agent。
- 通信拓扑当前按无向边构建对称矩阵。
- `tc` profile 按目标 IPv4 地址匹配出站流量；对称链路需要在两端都配置。
- 轮次调度目前是并发的事件驱动轮次，不是严格的离散事件网络模拟器。
- 应用事件与网络数据的跨层关联没有协议级因果 ID，只能基于 trace、Agent 身份和时间窗口推断。
